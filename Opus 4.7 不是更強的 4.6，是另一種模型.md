以下是您所選擇的 YouTube 影片《Opus 4.7 不是更強的 4.6，是另一種模型》為您整理且去除多餘空白斷詞的逐字稿：
大多數使用者升級 op4.7 之後，第一個感覺多半不是哇變強了，而是怎麼變難用了。
我看到一堆人在抱怨，社群上也有一堆人說同一份 prompt 塞進去輸出比以前還差。
但我自己連續用了一周下來，我認為事情不是大家在罵的那樣。
4.7 不是變笨，而是他把以前偷幫你做掉的工作丟回了你身上。
如果你還用舊方法跟他合作，當然會覺得他又貴又卡又不穩。
這支影片我想聊三件事：第一，為什麼 4.7 讓人覺得難用；第二，哪些痛點是真的，哪些是你其實有應對方法的；第三，我自己現在的工作流如何調整。
我先講一件事，因為我看到太多人搞混了。
4.7 出來之後的負評，本質上不是這個模型變笨了，是這個模型變得不好合作。
這兩件事完全不一樣：變笨是 capability 退步，benchmark 數字應該全面下滑；變難合作是 behavior，benchmark 可能還上升，但你跟他互動的體感變糟。
4.7 是後你看 Notion 團隊提出來的數字，他們測試 4.7 在多步驟任務上比 4.6 進步 14%，而且消耗了更少的 token，甚至 error 只剩三分之一。
Rocketon 在他們內部 SWE benchmark 上測出 4.7 解決的 production task 是 4.6 的三倍。
這些不是行銷噱頭，是真實商業場景實作下來的數據。
所以模型本身是有變強的，但一堆人還是在罵，原因是模型強的地方跟你日常感受到的地方是不一樣的。
你用他寫個東西、查個資料、升個段落、要他幫你理思緒，這些任務的強不是 SWE Bench 多兩位數，而是他有沒有先了解你的意圖，然後幫大多數人補完他們沒有講清楚的地方。
4.7 在這一塊改了行為，這就是為什麼很多人第一反應是難用，不是模型退步，是模型不再幫你補你沒講清楚的部分。
我實際使用了 OP4.7 後有一些心得跟大家分享：4.7 不是更強的 4.6，而是具備另一種行為模式的模型。
它最大的改變是更遵從你寫的字面意思，不替你腦補你沒說的部分，而這往往是過去讓中端使用者覺得好用的地方。
Anthropic 自己的文件寫得很清楚：模型不會推測你沒提的需求。
這句話翻譯成人話就是你說什麼，他做什麼。
你以前對 4.6 說：幫我把這篇文章變成三句話。
4.6 給你的可能是三句話，加上一個出題標題，加上重點，加上一個收尾，因為它推測你是要拿去發 Twitter 或者用在其他社群平台上，所以他順手幫你包好。
但你對 4.7 用一樣的提示詞，它真的就只會給你三句話。
這不是模型變笨，是他不再去執行你沒授權他做的事情。
這個改變的後果是兩面的。
好的一面是可控性變高，你寫 production prompt 最怕的就是模型自己加東西，今天加一句，明天加兩句，整個 pipeline 慢慢飄走，你也不知道是哪一步改的，4.7 把這件事情關掉了。
壞的一面是門檻變高。
你以前 prompt 寫得很爛，模型還是會給你看起來不錯的東西，因為他在替你補。
你今天 prompt 寫得很爛，4.7 就還你一個爛的東西。
所以從你的角度看，模型好像變難用了，但其實是你終於被迫看到了自己寫提示詞的水準。
我認為這個改變是對的方向。
Andrej Karpathy 自己說過，越強的模型越不需要你細節指揮，你應該給他成功標準，然後讓他自己想辦法。
4.7 是往這個方向走的第一個比較明顯的版本。
但這個方向對大多數的使用者很不友善，因為他把責任還給你了。
你要更清楚自己要什麼，你要更清楚什麼叫做好，這兩件事模型以前替你想，現在不想了。
這就是 4.7 整個故事的核心。
那我們來聊聊 OP4.7 三個讓人覺得難用的具體痛點。
第一個就是剛剛的內容延伸，這個模型更尊重指示，更少腦補，而這件事的影響範圍其實比你想像的還大。
最直接的就是寫作，你叫 4.7 幫你寫一段東西，它寫出來會有一種乾的感覺，沒有 4.6 那種自然的鋪陳、自然的 transition、自然的收尾。
因為 4.6 會替你判斷說這段東西最後應該收一下，中間這個概念應該加個比喻。
4.7 不會，你叫他寫一段，他就寫一段。
第二是 coding 相關的任務，同樣是重構某個 function，4.6 可能會順手把 input 整理好，把命名統一，把 deadcode 清掉。
因為他覺得按照邏輯來說，這些額外的東西你只是沒講，但你應該也想要。
但到了 4.7 身上，它真的就只重構那個 function，其他你沒說的通通不動。
第三是長任務執行，你用 Cloud code 跑一個多步驟任務，4.6 可能在某一步看到你還沒交代的東西，自己幫你決定。
4.7 不會，他走到那一步會停下來或是直接跳過。
第四個更微妙，你自己內心預期的成品改沒了。
你以前丟一個半成品給 4.6，它會還你一個感覺很完整的東西，雖然不一定真的對。
你今天丟一樣的半成品給 4.7，他還你一個半成品，雖然每一個他有處理的地方都更精準。
因為大多數人的 prompt 都不是 production level，大多數人的 prompt 是靠 4.6 自己補起來的，所以才會這麼多人詬病 4.7 的表現。
這件事情的解法不是把 prompt 寫得更長，這是大家最容易踩的坑。
你以為它變嚴格了，所以 prompt 要塞更多細節進去，其實不是。
Anthropic 自己也講了，越強的模型越不需要你下細節指令。
你該補的不是字數，是意圖，也就是要讓模型知道 user intention 是什麼，具體我會在影片後面再說明。
所以針對第一個痛點，結論是 4.7 的表現下降，通常是你的 prompt 不夠清楚，而 prompt 不夠清楚不是因為長短，而是意圖不夠清楚。
第二個痛點是費用。
4.7 比 4.6 貴，但它的標價跟 4.6 一模一樣，所以真正貴的其實是兩個東西。
第一個是 tokenizer，4.7 用的是新的 tokenizer，同樣一段文字、同樣一段 system prompt 餵進去之後算出來的 token 數會多上 35%。
這個數字還是保守值，社群上一位工程師叫做 Simon Willison，他實測 4.7 自己的 system prompt 在新 tokenizer 下會多出 1.46 倍的 token，超過 Anthropic 公佈的上限。
真實的 codebase、cloud、markdown、技術文件這類 coding agent 真的在吃的內容，獨立量測落在 1.29 到 1.47 倍之間。
這個意思是什麼？同樣一段 prompt，同樣一份 codebase，跑一次的成本就多 30% 到 47%。
每百萬 token 的費用沒漲，但你的帳單漲了。
第二個是 adaptive thinking，4.7 把 thinking budget 這個設計拿掉了。
以前你可以指定模型最多想多久，現在他自己決定。
高 effort 模式下，它會燒掉比預期更多的 output token。
這兩個東西加在一起，容錯空間變小了。
以前 prompt 寫得不夠好，跑壞一次大不了重來，token 也沒幾個錢。
今天你 prompt 寫得不夠好，跑壞一次的代價是雙倍，因為你燒了 30% 多的 input token，又燒了他自己決定要燒的 output token。
這就會給中斷用戶帶來一種怎麼更難用、更貴的感受。
以前你可以用便宜試錯的方式跟模型合作，現在不行了，現在每次出手前要想得更清楚一點。
這也是為什麼後面我會花很多時間講 review 流程跟模型分工，因為在 4.7 的世界裡，亂跑亂試的代價變高了。
第三個痛點我認為是最重要，但也最多人忽略的，4.7 的自我檢查能力不可靠。
因為他更敢直接給結論，更敢說我做完了，所以這個問題更明顯。
我自己跑過一組對照測試，我準備了幾份商業場景見的資料，故意在裡面埋了幾個陷阱，讓 4.7 跟 GPT5.4 各跑一遍，沒有人類介入，最後再讓兩邊互相 review。
我看到兩個值得你記住的現象：第一，4.7 有時候會宣稱我做完了，但實際上有些 task 他根本沒處理到，他寫的 report 跟他真正做的事情對不起來，他沒做但卻說他做了。
第二，我讓兩個模型互相 review，分數的形狀完全不一樣：Claude 自評高，GPT 給 Claude 的分數偏低；反過來 GPT 自評低，Claude 給 GPT 的分數偏高。
順帶一提，國外有一位獨立分析師做過更大規模的同類測試，他埋了更多陷阱，跑了更多檔案，結論跟我看到的基本一致，所以這不是我手上的個案，是模型本身的特性。
這個結果代表 OPUS 對自己過分樂觀，而 GPT 對自己過分嚴格。
你叫 OPUS 自己 review 自己，他會覺得自己很猛，但你叫他去 review 別人，他會嘴下留情。
而這對高風險任務是一場災難。
你寫一段 production code 叫 OPUS review，他說沒問題，你以為沒問題，其實有問題，他只是沒看出來或是看出來了但覺得沒事。
你做一份財務分析叫 OPUS 檢查，他說數字對，其實裡面有一個欄位是他幻覺出來的。
你寫一份對外的內容叫 OPUS 稿，他說 OK。
OPUS 4.7 的產出強，但它的自評能力弱，你不能再用跑一次自己 review 收工的方式做事，你需要外部的第二個眼睛，而那個第二個眼睛最好不是同一個模型。
講完三個痛點，我們進到解法。
我看到很多人的解法是降回 4.6，但我認為這是錯的。
不是因為 4.6 不能用，4.6 還是堪用，是因為這個方向是回不去的。
Anthropic 不會把這個產品方向改回去，下一個版本還會是這樣，你今天降回 4.6，下個月還是要面對一樣的事情。
真正的解法不是換回舊版，而是承認模型變了，然後你的 workflow、你的心態就得跟著一起調整。
模型變得更字面、更不腦補、更貴、更難自評，除非你不打算再使用 Anthropic 的模型，否則你的 workflow 就要針對這四個特質做改變。
所以接下來我會分享四個提升個人能力的方向。
這個頻道分享 AI 的使用策略、分享 AI 產業的最新趨勢、幫你去除噪音、建立正確的使用心態，如果你覺得內容對你有幫助，想請各位看官幫忙點個追蹤、按個讚，也分享給你的朋友們，那我們直接開始。
第一個是調整你的提示詞，但不是寫得更長，而是寫得更清楚。
最多人犯的錯是把 prompt 加長，原本 50 字的 prompt 變成 200 字，加一堆「請仔細」、「請完整」、「請考量」這類型的字詞在裡面。
這個方向是錯的，因為那些詞對 4.7 沒意義，他本來就會仔細、本來就會完整。
他需要的不是你催他努力，是你告訴他你要什麼，你想去達成什麼目標。
真正該補的東西其實就四個：第一，這個任務的目的是什麼？第二，這個產出要給誰看？第三，好的產出是什麼樣子的？第四，一定要避開的錯誤有哪些？
舉個簡單的例子，你以前寫「幫我把這篇文章寫成三句話」，比較常見的方法是改成「幫我把這篇文章寫成三句話，每句話要 punchy，重點要明確，用粗體標出關鍵字，最後加一個 takeaway」。
這樣稍微好了一點，但還是不對，因為這個 prompt 還是在描述表面的細節，不是使用者的意圖。
正確的做法應該是：「幫我把這篇文章變成三句話，目的是發在我的 Twitter，我的讀者是 AI 的 Builder，他們時間很少，如果第一句沒讓他們看到重點就會直接滑掉。」
這就是所謂的 success criteria，你給模型場景、讀者、判斷標準，他自己會決定要不要 punchy、要不要 highlight、要不要 takeaway。
這就是 AI 大佬 Andrej Karpathy 說過的，不要再告訴模型怎麼做，告訴他什麼是做好的。
這是最便宜效益最大的升級，你不需要新工具，你不需要換模型，你只需要在每次 prompt 開頭多寫兩三句意圖。
做到這件事後，你會發現模型的表現大幅提升，但前提是你願意花那 30 秒先想清楚你要什麼。
這 30 秒就是 4.7 在逼你不要偷懶，想清楚你到底想要什麼。
老話一句，如果這件事對你來說很難，你可以問問自己，當我身為需求提出方，我都講不清楚我自己想做什麼，以及想這樣做的原因，到底要怎麼把這個任務委派給你的 agent 呢？
第二個調整方向是 cross model review，比方說讓 codex review Claude。
我先講為什麼這件事是必要的，不是 nice to have。
回到剛剛那個測試的結論：Claude 自評高，GPT 給他的分數偏低；GPT 自評低，Claude 給他的分數偏高。
同樣一份產出，當事模型跟旁觀模型給出完全不同的分數。
你叫一個學生改自己的考卷，他多半改不出錯；你叫他改別人的考卷，他多半改得出錯。
模型也一樣，他對自己的產出有偏誤，要么太樂觀，要么太悲觀，但他對另一個模型的產出，這個偏誤就消失了。
所以 cross model review 不是學術上的玄學，是真的在解決使用者看到的，而且經常發生的 failure patterns。
這邊講一下我自己怎麼用。
我每天主要用兩個 coding agent，分別是 Claude code 跟 codex client。
過去半年我用法的變化是這樣：以前我會挑一個用到爽，現在我幾乎所有重要任務都會走兩段：第一段叫 producer，第二段叫 reviewer。
producer 寫第一版，reviewer 看第一版。
具體而言，誰當 producer、誰當 reviewer 取決於你的任務。
寫架構、寫設計、寫長 context 的任務，我傾向 Claude 當 producer，codex 當 reviewer，因為 Claude 4.7 在多步驟跟長 context 上比較強，但它會 oversell，所以讓 codex 當 reviewer 剛好抓那個 oversell。
寫純執行、寫精準 patch、寫一塊隔離的 function，我傾向 codex 當 producer，Claude 當 reviewer，因為 GPT 在精準執行上更直接，但它會 undersell，會自己挑毛病挑到偏，讓 Claude 當 reviewer 平衡一下。
具體做法有兩種：第一種最簡單是 copy paste，我讓 Claude 寫完，把整個 code 連同原始任務丟到 codex，告訴他請以一個資深工程師的角度 review 這個 PR，找出邏輯錯誤、邊界條件跟原始需求對不上的地方。
第二種比較進階是讓兩個 agent 直接互相對話，可以直接請 Claude 呼叫 codex 或是反過來。
簡單來說就是讓主 agent 完成任務之後，直接請另一個模型 review，把 review 結果再交回主 agent 修，省去複製貼上的時間。
你只需要做一次就會有感，你會看到 codex 抓出 Claude 自己 review 自己抓不到的東西。
這個價值不是錦上添花，是阻止你把幻覺當成最後產出交付出去。
Claude 說我 review 過了是真的有 review，它就是看不到自己漏掉的東西。
我認為現在真正穩的 workflow 不是找最強的單一模型，而是設計能互相驗證的多模型流程。
這件事用範圍比你想像的廣。
coding 是最明顯的場景，你寫 code 另一個模型 review code，這個流程建一次受用一輩子。
研究摘要也適用，你叫 Claude 整理一份論文摘要，再叫 codex 比對原文挑錯。
論點結構也適用，你叫一個模型寫論述，再叫另一個模型挑邏輯漏洞。
財務跟資料分析尤其適用，一個小數點放錯位置、一個關鍵數字算錯，這些都是模型自己 review 不到的，但如果讓另一個模型再過一遍，通常都會被攔下來。
對外內容也適用，重要的對外稿，永遠讓另一個模型過一次。
做這件事最大的成本不是 token，是你願不願意建這個習慣。
但一旦你建了，你會發現你 ship 出去的東西品質直接提高一個檔次。
第三個是 model routing。
4.7 不是萬能的，它有強項，也有明確的弱項。
你不能什麼都丟給它，那叫亂用。
4.7 有幾個很明確的強項：第一，多步驟需要長執行的 agentic 任務。
SWE Bench Verified 從 80.8% 拉到 87.6%，Cursor Bench 從 58% 拉到 70%，MCP Atlas 多工具編排從 75.8 拉到 77.3%。
第二，深度知識工作。
GDPV 這個指標講白話就是測模型做專業知識工作的能力，分析報告、整理這類事情。
在這個指標上 4.7 拿 1753 分，GPT5.4 拿 1674 分，Gemini 3.1 Pro 才 1314 分，差距非常大。
幾家做企業 AI 的公司也都驗證了同一個方向：做資料分析平台的 HX 把 4.7 評為他們測過最強的模型，光是金融分析這一項就從 0.767 拉到 0.813；做法律 AI 的 Harvey 在他們的法律專業測試上拿到 90.9%；做資料平台的 data bricks 在他們的 office 工作測試裡 error 少了 21%。
第三，視覺辨識。
獨立的視覺測試裡 4.7 的識別準確率從 54.5% 一路拉到 98.5%。
意思是它現在能讀截圖上的小字，能看懂密集的圖表，這在以前的模型上幾乎是做不到的。
從這幾個維度來看的話，4.7 是你現在能拿到最強的工具。
但它有兩個明確的弱項：第一，web research。
browscom 是專門測模型上網搜尋整理資料能力的 benchmark，4.7 從 83.7 掉到 79.3，GPT5.4 Pro benchmark 領先 10 分，Gemini 3.1 Pro 領先 6 分。
如果你的 agent 主要在做網頁搜尋、reading、整理，這塊用 GPT 或 Gemini 表現會比較好。
第二，terminal 任務。
Terminal bench 測模型在命令環境下下 command 串流程的能力，4.7 在這個 benchmark 上落後 GPT5.4 將近 6 分。
如果你的 agent 主要住在 terminal 裡下 command，這塊也該分流。
所以模型分工不是哲學，是你身為 AI 原生工作者，在了解 benchmark 之後的工程決策。
我自己現在的分工大概長這樣：深度 coding 跟長任務主用 Claude 4.7；知識型文件的彙整跟處理也是 Claude 4.7；純 coding 或者 terminal 任務主用 codex；web research、整理網頁資料主用 Gemini 3.1 Pro 或 GPT。
最最最重要的是 review 一定要跨模型，不能同一個。
我給你幾個更具體的 case，你比較有感：寫一個跨多個檔案，需要看完整 codebase 才能改的 refactor，這種任務 Claude 點起來比較穩，因為它的 multistep assistance 比 codex 強，跑到一半不會自己宣布完工。
寫一個小工具、一個 utility script、一個 isolated function，這種任務丟給 codex 比較划算，因為 codex 又便宜又快，task 範圍清楚的時候，它的精準度其實夠用，沒必要燒 4.7 的 token。
整理一份 30 篇論文摘要做 literature review，我會先讓 Gemini 3.1 Pro 網頁抓取跟初步整理，再讓 Claude 4.7 把所有摘要整合成一份結構化文件，因為知識整合是 4.7 的強項。
這個分工不是固定的，下一個模型出來很可能就要隨時調整。
你不用特別關注這些東西，因為我只要有這類型的使用心得，都會先幫你整理好。
高手不是忠於一個模型，而是知道哪個工作該給誰做。
第四個升級，這個是最難的，但也是最重要的，就是把 review 變成你工作流裡的固定流程，而不是看心情去 review。
大多數人的 review 流程多半是憑感覺，你用完模型看眼出，覺得嗯看起來 OK 然後就交出去。
但不用我說你也知道，憑感覺 review 就是賭博。
所以你要把 review 變成一個有結構的流程，不是一個動作。
最簡單暴力的版本拆成三步：第一步主 agent 產出，第二步 cross model review，第三步人類 final pass。
把這三步寫進你的 SOP，每次重要任務都跑一遍。
這個流程的好處不是他一定不會錯，是他把錯誤擋在模型交付給你之前，而不是之後才被你抓到。
這件事最值得投資的地方不在 prompt 在 infrastructure。
你應該花時間建一個 cross model review 的 template 或 subagent，一個固定的 review checklist，一個自動化跑這個 pipeline 的小工具。
這些東西只要建起來一次，之後的每一天都會給你帶來效益。
舉一個我自己的例子，我會請 Claude 先寫一份實作計劃，計劃寫完，我直接讓 Claude 在 terminal 裡呼叫一個 codex 對這份計劃做至少兩輪 review。
為什麼是兩輪？因為 review 可以無限做下去，你永遠可以雞蛋裡挑骨頭，所以你一定要自己設一個收手點。
我抓兩輪的經驗是兩輪通常已經夠把真正的大問題抓出來，再多輪編輯效益就掉得很快。
這支影片講的是大方向，具體技術上怎麼接、prompt 長什麼樣，可以看影片下方資訊欄的文件。
這也回到一個很重要的使用心態：你不只是在用模型，你是在建一套自己的 AI 生產系統。
模型會換、會升級、會被取代，但你的系統如果建得好，下個模型來，你只要換掉中間那一塊，整個 pipeline 還在。
這件事情應該是 4.7 這次升級最重要的 takeaway：不要把自己綁在某一個模型上，而是設計一套 workflow，抱著正確的心態，跟著模型一起升級。
最後總結一下，4.7 不是一個更強的 4.6，它是一個調性有點不同的模型。
他更遵循指示，更不會腦補，更貴，更敢直接下結論，同時也因為更有自信，所以自我審核反而出現問題。
這個改變或許對大多數使用者很不友善，因為他把以前模型替你做的事還回到你身上，但這個方向是回不去的。
你要學會，Anthropic 的產品總負責人 Mike Krieger 本身是 Instagram 的共同創辦人，他在 4.7 發表前三天才從 Figma 的董事會退下來。
Figma 是做設計工具的，Anthropic 同一個禮拜就發了一個叫 Claude design 的產品，直接跟 Figma 打對台。
所以 Krieger 退下來不是巧合，是利益迴避。
Anthropic 不是在做一個讓你聊天爽的 chatbot，而是在做一個能取代專業工作流的 Enterprise 2 公司。
公司現在估值 8000 億美金，四月年化營收 300 億美金，下一個模型只會更精準、更尊重指令、更敢直接做事。
降回 4.6 是緩兵之計，你下個月還是要面對這件事。
真正的解法是個人心態跟做事方法論的調整。
簡單講就四件事：第一，prompt 不要寫更長，要寫得更清楚，補的是意圖，不是字數。
第二，高風險任務做 cross model review，讓 codex review Claude 或者反過來。
第三，模型分工，深度 coding 跟知識工作給 4.7，web research 跟 terminal 任務給 GPT 或 Gemini。
第四，把 review 流程化寫進 SOP，寫成 subagent，寫成 template，同時建立你自己的 QA checklist。
以前模型替你做的判斷，現在你要自己做；以前模型替你補的細節，現在你要自己補；以前模型替你檢查的東西，現在你要建另一個流程檢查。
這聽起來很煩，但那個丟一句 prompt 就拿到完美結果的時代，本來就是假象，4.7 只是把這個假象戳破。
AI 不是通靈法師，它是一個聰明到爆，但你得點出方向的工作夥伴。
不是 4.7 難用，是你不能再用舊方法用它。
如果你最近也被 4.7 搞到很躁，我給你的建議是先回頭看你現在的 workflow 有沒有跟著升級 prompt 寫法、review 流程、模型分工。

以下為您梳理這段內容的核心邏輯與架構級洞見：

### 核心洞察：從「依賴工具」到「掌握抽象思維」
影片的核心觀點完美印證了一件事：**在 AI 驅動的開發中，掌握「抽象與思考的底層邏輯」遠比追逐特定、轉瞬即逝的提示詞技巧更為重要。**
*   **行為模式的轉變：** 較新的高階模型不再擅自「腦補」使用者的模糊需求，而是嚴格遵循字面的指示。這對習慣依賴模型通靈的用戶是痛點，但對追求精準控制、有清晰系統架構的開發者而言，反而大幅提升了輸出的可控性。
*   **意圖為王：** 解決輸出品質下降的方法，並非無意義地堆砌提示詞字數，而是清晰定義「成功標準 (Success Criteria)」、目標與邊界條件。你需要將模型視為嚴格的執行者，提供清晰的商業邏輯與意圖，而非期待它替你完成思考。

### 系統級架構：多智能體協同 (Multi-Agent Orchestration) 的必然性
影片中提出的解法，本質上就是在呼籲建立多智能體協同的編排架構。這與打造自動化工作流、去中心化檢驗的開發理念高度契合：
1.  **Cross-Model Review (跨模型審查)：** 模型普遍存在「自我審查盲區」，例如 Opus 4.7 常表現出過度自信。將生產 (Producer) 與審查 (Reviewer) 的角色拆分，交由不同模型執行（例如讓擅長長文本與規劃的模型撰寫草稿或架構，讓擅長精確邏輯的模型負責挑錯與 Review），能有效攔截幻覺。
2.  **Model Routing (模型分流)：** 沒有單一的完美模型。複雜的長任務與深度架構思考可以交給表現最好的模型，而單純的函式重構、終端機指令或網頁檢索，則應分流給速度更快或具備特定優勢的 AI。

具體而言，誰當 producer、誰當 reviewer 取決於你的任務。
寫架構、寫設計、寫長 context 的任務，我傾向 Claude 當 producer，codex 當 reviewer，因為 Claude 4.7 在多步驟跟長 context 上比較強，但它會 oversell，所以讓 codex 當 reviewer 剛好抓那個 oversell。
寫純執行、寫精準 patch、寫一塊隔離的 function，我傾向 codex 當 producer，Claude 當 reviewer，因為 GPT 在精準執行上更直接，但它會 undersell，會自己挑毛病挑到偏，讓 Claude 當 reviewer 平衡一下。
具體做法有兩種：第一種最簡單是 copy paste，我讓 Claude 寫完，把整個 code 連同原始任務丟到 codex，告訴他請以一個資深工程師的角度 review 這個 PR，找出邏輯錯誤、邊界條件跟原始需求對不上的地方。
第二種比較進階是讓兩個 agent 直接互相對話，可以直接請 Claude 呼叫 codex 或是反過來。
簡單來說就是讓主 agent 完成任務之後，直接請另一個模型 review，把 review 結果再交回主 agent 修，省去複製貼上的時間。
你只需要做一次就會有感，你會看到 codex 抓出 Claude 自己 review 自己抓不到的東西。
這個價值不是錦上添花，是阻止你把幻覺當成最後產出交付出去。
Claude 說我 review 過了是真的有 review，它就是看不到自己漏掉的東西。
我認為現在真正穩的 workflow 不是找最強的單一模型，而是設計能互相驗證的多模型流程。
這件事用範圍比你想像的廣。
coding 是最明顯的場景，你寫 code 另一個模型 review code，這個流程建一次受用一輩子。
研究摘要也適用，你叫 Claude 整理一份論文摘要，再叫 codex 比對原文挑錯。
論點結構也適用，你叫一個模型寫論述，再叫另一個模型挑邏輯漏洞。
財務跟資料分析尤其適用，一個小數點放錯位置、一個關鍵數字算錯，這些都是模型自己 review 不到的，但如果讓另一個模型再過一遍，通常都會被攔下來。
對外內容也適用，重要的對外稿，永遠讓另一個模型過一次。
做這件事最大的成本不是 token，是你願不願意建這個習慣。
但一旦你建了，你會發現你 ship 出去的東西品質直接提高一個檔次。
第三個是 model routing。
4.7 不是萬能的，它有強項，也有明確的弱項。
你不能什麼都丟給它，那叫亂用。
4.7 有幾個很明確的強項：第一，多步驟需要長執行的 agentic 任務。
SWE Bench Verified 從 80.8% 拉到 87.6%，Cursor Bench 從 58% 拉到 70%，MCP Atlas 多工具編排從 75.8 拉到 77.3%。
第二，深度知識工作。
GDPV 這個指標講白話就是測模型做專業知識工作的能力，分析報告、整理這類事情。
在這個指標上 4.7 拿 1753 分，GPT5.4 拿 1674 分，Gemini 3.1 Pro 才 1314 分，差距非常大。
幾家做企業 AI 的公司也都驗證了同一個方向：做資料分析平台的 HX 把 4.7 評為他們測過最強的模型，光是金融分析這一項就從 0.767 拉到 0.813；做法律 AI 的 Harvey 在他們的法律專業測試上拿到 90.9%；做資料平台的 data bricks 在他們的 office 工作測試裡 error 少了 21%。
第三，視覺辨識。
獨立的視覺測試裡 4.7 的識別準確率從 54.5% 一路拉到 98.5%。
意思是它現在能讀截圖上的小字，能看懂密集的圖表，這在以前的模型上幾乎是做不到的。
從這幾個維度來看的話，4.7 是你現在能拿到最強的工具。
但它有兩個明確的弱項：第一，web research。
browscom 是專門測模型上網搜尋整理資料能力的 benchmark，4.7 從 83.7 掉到 79.3，GPT5.4 Pro benchmark 領先 10 分，Gemini 3.1 Pro 領先 6 分。
如果你的 agent 主要在做網頁搜尋、reading、整理，這塊用 GPT 或 Gemini 表現會比較好。
第二，terminal 任務。
Terminal bench 測模型在命令環境下下 command 串流程的能力，4.7 在這個 benchmark 上落後 GPT5.4 將近 6 分。
如果你的 agent 主要住在 terminal 裡下 command，這塊也該分流。
所以模型分工不是哲學，是你身為 AI 原生工作者，在了解 benchmark 之後的工程決策。
我自己現在的分工大概長這樣：深度 coding 跟長任務主用 Claude 4.7；知識型文件的彙整跟處理也是 Claude 4.7；純 coding 或者 terminal 任務主用 codex；web research、整理網頁資料主用 Gemini 3.1 Pro 或 GPT。
最最最重要的是 review 一定要跨模型，不能同一個。
我給你幾個更具體的 case，你比較有感：寫一個跨多個檔案，需要看完整 codebase 才能改的 refactor，這種任務 Claude 點起來比較穩，因為它的 multistep assistance 比 codex 強，跑到一半不會自己宣布完工。
寫一個小工具、一個 utility script、一個 isolated function，這種任務丟給 codex 比較划算，因為 codex 又便宜又快，task 範圍清楚的時候，它的精準度其實夠用，沒必要燒 4.7 的 token。
整理一份 30 篇論文摘要做 literature review，我會先讓 Gemini 3.1 Pro 網頁抓取跟初步整理，再讓 Claude 4.7 把所有摘要整合成一份結構化文件，因為知識整合是 4.7 的強項。
這個分工不是固定的，下一個模型出來很可能就要隨時調整。
你不用特別關注這些東西，因為我只要有這類型的使用心得，都會先幫你整理好。
高手不是忠於一個模型，而是知道哪個工作該給誰做。
第四個升級，這個是最難的，但也是最重要的，就是把 review 變成你工作流裡的固定流程，而不是看心情去 review。
大多數人的 review 流程多半是憑感覺，你用完模型看眼出，覺得嗯看起來 OK 然後就交出去。
但不用我說你也知道，憑感覺 review 就是賭博。
所以你要把 review 變成一個有結構的流程，不是一個動作。
最簡單暴力的版本拆成三步：第一步主 agent 產出，第二步 cross model review，第三步人類 final pass。
把這三步寫進你的 SOP，每次重要任務都跑一遍。
這個流程的好處不是他一定不會錯，是他把錯誤擋在模型交付給你之前，而不是之後才被你抓到。
這件事最值得投資的地方不在 prompt 在 infrastructure。
你應該花時間建一個 cross model review 的 template 或 subagent，一個固定的 review checklist，一個自動化跑這個 pipeline 的小工具。
這些東西只要建起來一次，之後的每一天都會給你帶來效益。
舉一個我自己的例子，我會請 Claude 先寫一份實作計劃，計劃寫完，我直接讓 Claude 在 terminal 裡呼叫一個 codex 對這份計劃做至少兩輪 review。
為什麼是兩輪？因為 review 可以無限做下去，你永遠可以雞蛋裡挑骨頭，所以你一定要自己設一個收手點。
我抓兩輪的經驗是兩輪通常已經夠把真正的大問題抓出來，再多輪編輯效益就掉得很快。
這支影片講的是大方向，具體技術上怎麼接、prompt 長什麼樣，可以看影片下方資訊欄的文件。
這也回到一個很重要的使用心態：你不只是在用模型，你是在建一套自己的 AI 生產系統。
模型會換、會升級、會被取代，但你的系統如果建得好，下個模型來，你只要換掉中間那一塊，整個 pipeline 還在。
這件事情應該是 4.7 這次升級最重要的 takeaway：不要把自己綁在某一個模型上，而是設計一套 workflow，抱著正確的心態，跟著模型一起升級。
最後總結一下，4.7 不是一個更強的 4.6，它是一個調性有點不同的模型。
他更遵循指示，更不會腦補，更貴，更敢直接下結論，同時也因為更有自信，所以自我審核反而出現問題。
這個改變或許對大多數使用者很不友善，因為他把以前模型替你做的事還回到你身上，但這個方向是回不去的。
你要學會，Anthropic 的產品總負責人 Mike Krieger 本身是 Instagram 的共同創辦人，他在 4.7 發表前三天才從 Figma 的董事會退下來。
Figma 是做設計工具的，Anthropic 同一個禮拜就發了一個叫 Claude design 的產品，直接跟 Figma 打對台。
所以 Krieger 退下來不是巧合，是利益迴避。
Anthropic 不是在做一個讓你聊天爽的 chatbot，而是在做一個能取代專業工作流的 Enterprise 2 公司。
公司現在估值 8000 億美金，四月年化營收 300 億美金，下一個模型只會更精準、更尊重指令、更敢直接做事。
降回 4.6 是緩兵之計，你下個月還是要面對這件事。
真正的解法是個人心態跟做事方法論的調整。
簡單講就四件事：第一，prompt 不要寫更長，要寫得更清楚，補的是意圖，不是字數。
第二，高風險任務做 cross model review，讓 codex review Claude 或者反過來。
第三，模型分工，深度 coding 跟知識工作給 4.7，web research 跟 terminal 任務給 GPT 或 Gemini。
第四，把 review 流程化寫進 SOP，寫成 subagent，寫成 template，同時建立你自己的 QA checklist。
以前模型替你做的判斷，現在你要自己做；以前模型替你補的細節，現在你要自己補；以前模型替你檢查的東西，現在你要建另一個流程檢查。
這聽起來很煩，但那個丟一句 prompt 就拿到完美結果的時代，本來就是假象，4.7 只是把這個假象戳破。
AI 不是通靈法師，它是一個聰明到爆，但你得點出方向的工作夥伴。
不是 4.7 難用，是你不能再用舊方法用它。
如果你最近也被 4.7 搞到很躁，我給你的建議是先回頭看你現在的 workflow 有沒有跟著升級 prompt 寫法、review 流程、模型分工。

Opus 4.7 不是更強的 4.6，是另一種模型

這是 roadmap signal。

## 2. 定價結構

企業 plan：

- seat license

- admin features
    

個人 plan 多數係：

- 月費有限額
    
- 高峰期限制
    
- 較低優先權
    

收入模型明顯偏 B2B。

## 3. 回答風格改變

模型愈來愈：

- structured
    
- safer
    
- concise
    
- deterministic
    
- policy aligned
    

呢啲都係企業部署友好特徵。

# 四、點解你會覺得變笨？

因為你量度嘅指標同企業唔同。

## 個人用戶量度：

- 會唔會心領神會
    
- 幫唔幫我補完想法
    
- 有冇驚喜
    
- 似唔似聰明伙伴
    

## 企業量度：

- 錯誤率
    
- 每任務成本
    
- 可預測性
    
- 可追蹤
    
- 安全性
    
- integration ease
    

所以同一模型：

個人：退步  
企業：升級

# 五、如何真正驗證，而唔係靠講

你自己可以做 A/B test。

## 測試集 1：Consumer 體感任務

- 模糊 brainstorm
    
- 情緒支持
    
- 幫我諗創意
    
- 半成品變完整文案
    

## 測試集 2：Enterprise 任務

- JSON extraction
    
- spec to code
    
- contract summarization
    
- structured comparison
    
- spreadsheet logic
    
- deterministic rewrite
    

如果模型在 2 明顯進步，在 1 未必進步甚至下降，假說成立。

# 六、我唔會誇張講「一定冇變笨」

更精準講法係：

> 某些舊版在 consumer chat 體感上更討喜。  
> 新版在可部署能力上更強。

這不是矛盾。

好似：

- 跑車操控爽，但載貨差
    
- 商用 van 唔刺激，但實用高
    

你話跑車「更好玩」係真。  
企業話 van「更有價值」都真。

# 七、最硬核證據：資本流向

如果市場認為聊天陪伴最值錢，資金會集中：

- personality AI
    
- companion AI
    
- entertainment bots
    

但現時最大估值與投資集中係：

- enterprise AI stack
    
- coding AI
    
- productivity AI
    
- infra / agent systems
    

錢通常最誠實。

# 八、結論

要證明我講法，不係靠一句話，而係三點同時成立：

1. 企業任務 benchmark 持續上升
    
2. 產品 roadmap 明顯偏部署與管理
    
3. 個人聊天體感未同步提升甚至下降
    

如果三點成立，就支持：

> 模型未必變笨，而係優化方向轉向企業價值函數。
